# M005 S07 local UAT walkthrough

This walkthrough is the executable local proof for the final learner flow in MiraLingo. It covers the logged-out landing page, learner registration/login, menu navigation, practice modes, typed answers, reveal flow, audio diagnostics, analytics, settings persistence, Iconify behavior, and confirmed account deletion.

Deterministic backend/frontend tests remain the source of truth for API and storage contracts. This document exists because the package does not currently ship a Playwright or Cypress dependency, so browser-tool/manual UAT is the durable acceptance path.

## Scope and limits

- Run everything from the repository root: `packages/webapp/` depends on in-repo source packages.
- Keep all work inside this repository.
- This is a single-learner, local-development flow.
- Browser audio still needs manual observation. The browser can prove that the button, status text, headers, and diagnostics are correct, but actual audible playback may depend on your local audio device/browser policy.
- Iconify live matching depends on network access to the public Iconify API. A visible fallback/offline state is acceptable and should be recorded as a pass when the UI stays safe and informative.
- Use a throwaway SQLite path if you want isolated state between runs.

## Local startup

### Optional throwaway database

Use a dedicated SQLite file when reproducing persistence or deletion behavior so an older browser session does not reuse durable state:

```bash
export MIRALINGO_DATABASE_PATH=/tmp/m005-s07-uat.sqlite3
```

If you do not set `MIRALINGO_DATABASE_PATH`, the backend uses its default local SQLite path.

### Backend

Start the backend from the repository root in one terminal:

```bash
PYTHONPATH=packages/webapp/src uvicorn mirad_webapp.api:create_app --factory --reload
```

Recommended full local source path when you want the real translator/TTS-backed flow rather than backend-only smoke behavior:

```bash
MIRALINGO_DATABASE_PATH=${MIRALINGO_DATABASE_PATH:-/tmp/m005-s07-uat.sqlite3} \
PYTHONPATH=packages/webapp/src:packages/tts/src:packages/translator/src \
uvicorn mirad_webapp.api:create_app --factory --reload
```

Expected backend surfaces:

- App listens on `http://127.0.0.1:8000`
- Logged-out `GET /auth/current-user` returns HTTP `401` with password-free JSON
- Logged-out `GET /settings` returns HTTP `401` with `phase: settings_get`
- Logged-out `GET /practice/progress` returns HTTP `401` with `phase: practice_progress`

### Frontend

Start the frontend from the repository root in a second terminal:

```bash
npm --prefix packages/webapp/frontend run dev
```

Expected frontend surface:

- Vite serves the app, usually on `http://127.0.0.1:5173`
- The browser shows the logged-out landing page first

## Quick preflight diagnostics

Run these before opening the browser if startup looks suspicious:

```bash
curl -i http://127.0.0.1:8000/auth/current-user
curl -i http://127.0.0.1:8000/settings
curl -i http://127.0.0.1:8000/practice/progress
```

Expected logged-out behavior:

- `/auth/current-user` → `401` with `{"authenticated": false, "user": null}`
- `/settings` → `401` with `error: unauthenticated`, `phase: settings_get`
- `/practice/progress` → `401` with `error: unauthenticated`, `phase: practice_progress`

Do not treat these logged-out responses as failures.

## Browser UAT walkthrough

Record screenshots or browser-tool evidence at each major checkpoint.

### 1. Logged-out landing page and public links

1. Open the frontend URL.
2. Confirm the page shows `Welcome to MiraLingo`.
3. Confirm the public learning/landing content includes the local MiraLingo messaging and the external `Wikibooks grammar` link.
4. Click the `Wikibooks grammar` link and confirm it opens externally.
5. Return to the app tab.

Capture:

- Visible landing heading
- The presence of the `Wikibooks grammar` link
- Any `role="status"` or `role="alert"` auth message shown while the app checks the session

### 2. Register a learner and land on the authenticated menu

1. Register a fresh learner account from the landing form.
2. Use a non-admin learner username so settings persistence and account deletion are safe to verify.
3. Confirm registration succeeds and the app transitions to the authenticated shell.
4. Confirm the menu shows these entries:
   - `Continue Practice`
   - `Revision`
   - `Build Vocabulary`
   - `Analytics`
   - `Settings`
   - `Log Out`

Capture:

- Successful transition from logged-out landing to authenticated menu
- Browser-visible success/status text if present
- `GET /auth/current-user` showing the learner identity without password/hash/salt

Suggested API check after registration/login:

```bash
curl -i http://127.0.0.1:8000/auth/current-user
```

### 3. Continue Practice mode

1. Click `Continue Practice`.
2. Confirm the app loads a mixed queue rather than staying on the menu.
3. Confirm the current card area shows card metadata such as card type and queue status.
4. Confirm the queue can contain both `word` and `phrase` cards.
5. Confirm the section behaves like the main practice path, not analytics or settings.

Capture:

- Visible current card
- Any queue-loading `role="status"` text
- `GET /practice/progress` before answering anything, if you want a baseline

### 4. Typed correct answer

1. In `Continue Practice`, type the exact expected answer for the current card.
2. Submit the typed answer.
3. Confirm the result announces success using visible status text such as `Correct`.
4. Confirm the card advances or the queue refreshes.
5. Confirm the answer result surface includes backend-truth fields such as expected/submitted answer context rather than a frontend-only guess.

Capture:

- The typed input before submit
- The visible correct-answer status
- Updated analytics/progress evidence from `/practice/progress`

Suggested API check:

```bash
curl -i http://127.0.0.1:8000/practice/progress
```

### 5. Wrong typed answer

1. On a later card, enter an intentionally wrong typed answer.
2. Submit it.
3. Confirm the UI shows a visible not-correct result rather than failing silently.
4. Confirm the response does not expose secrets, stack traces, password hashes, or salts.
5. Confirm analytics/progress changes after the wrong answer.

Capture:

- Wrong typed input
- Visible failure/result text
- Updated `/practice/progress` payload or browser-visible analytics status

### 6. Blank answer negative check

1. Focus the typed answer box.
2. Submit a blank or whitespace-only answer.
3. Confirm the UI handles it safely: no crash, no secret leak, and a visible error or blocked submit path.
4. Confirm any resulting feedback appears via `role="alert"` or `role="status"`.

Capture:

- What happened after blank submit
- Any visible error/status copy
- Whether the queue stayed stable

### 7. Give Up reveal flow

1. On the next card, click `Give Up`.
2. Confirm the app records the answer as wrong.
3. Confirm the UI reveals the expected answer.
4. Confirm the reveal messaging states that the correct answer is shown below.
5. Confirm this path also updates progress/analytics.

Capture:

- The visible reveal
- The visible result/error/status text
- Updated `/practice/progress`

### 8. Revision mode checks

1. Return to the menu if needed.
2. Open `Revision`.
3. Confirm the queue loads in revision mode.
4. If no revision cards are available yet, confirm the UI surfaces an understandable empty-state rather than an app error.

Capture:

- Queue/status text

### 9. Build Vocabulary mode checks

1. Return to the menu if needed.
2. Open `Build Vocabulary`.

### 10. Audio playback and diagnostics

1. On a card with audio available, click `Hear Mirad answer`.
2. If audio plays, confirm the browser behaves normally and the UI enters a loading/playing status first.
3. If local audio runtime is unavailable, confirm the UI shows a structured unavailable diagnostic instead of a generic crash.
4. Confirm the status/error surface uses `role="status"` for non-fatal updates and `role="alert"` for error/unavailable states.
5. Confirm saved learner speed is reflected in the audio experience or diagnostic text.

Expected acceptable outcomes:

- Success path: WAV response and audible playback
- Acceptable local fallback path: structured unavailable state such as `mbrola_unavailable`, `mbrola_voice_unavailable`, or a safe backend-unavailable diagnostic

Suggested API/header check:


Capture:

- Visible audio button state
- Visible audio diagnostic/status text

### 13. Iconify matched icon and forced fallback/offline behavior

1. In a practice mode, note the icon area for the current card.
2. Confirm the UI exposes `Iconify status: ...` text.
3. With normal network access, confirm the app can show a matched icon image for at least one card.
4. Then force an offline or blocked-network condition for Iconify if practical.
5. Refresh or move to another card.
6. Confirm the UI falls back safely instead of breaking layout or hiding diagnostics.
7. Treat the fallback/offline state as an acceptable pass if the app still shows clear status text and a safe non-broken visual result.

1. Register or log in as a non-admin learner account that is safe to delete.
2. Open `Settings`.

7. Confirm the UI reports deletion success.
8. Confirm the current session is torn down immediately.

- Logged-out state after deletion

## Automation limits

- Browser automation can prove navigation, visible text, network calls, and error surfaces.
- Audible success still may require manual observation because browsers can gate playback by device/browser policy.
- Iconify live quality depends on public network access and the external service. A fallback or offline state is acceptable when the UI keeps a visible safe diagnostic and no broken HTML/script behavior appears.
- This repository does not currently include Playwright/Cypress as the acceptance runner for this flow, so manual or browser-tool walkthrough evidence is the intended proof layer above deterministic tests.

## Failure modes and where to look

### Backend port conflict

Symptoms:

- Uvicorn fails to start
- Frontend shows auth/connectivity failure

Look at:

- Backend terminal output
- `curl http://127.0.0.1:8000/auth/current-user`
- Browser-visible auth error or landing-page alert

Suggested action:

- Free port `8000` or restart the backend on an unused port and update the frontend proxy/config if needed.

### Frontend port conflict

Symptoms:

- Vite fails to bind
- Browser cannot open the expected local URL

Look at:

- Frontend terminal output
- Vite startup URL

Suggested action:

- Free the current Vite port or use the printed alternate port.

### Missing MBROLA

Symptoms:

- Clicking `Hear Mirad answer` yields unavailable/error diagnostics
- Audio endpoint returns structured `503` JSON

Look at:

- Audio button status text
- Browser network response for `/practice/audio/<card-id>`
- Response headers/body

Suggested action:

- Install MBROLA locally or accept the structured unavailable diagnostic as the expected fallback for this UAT.

### Iconify timeout or offline behavior

Symptoms:

- No remote icon match loads
- UI reports fallback or timeout state

Look at:

- Visible `Iconify status: ...`
- Browser network logs for `api.iconify.design`
- Offline/devtools network settings

Suggested action:

- Record the fallback state as acceptable if UI remains informative and stable.

### Malformed API response

Symptoms:

- Browser section fails to render expected data
- Visible alert/status shows a safe error

Look at:

- Browser network inspector for `/auth/current-user`, `/settings`, `/practice/progress`, or `/practice/audio/<card-id>`
- Backend terminal output
- Visible `role="alert"` error text

Suggested action:

- Save the response payload and visible error copy, then compare with deterministic tests.

### Stale browser session

Symptoms:

- The browser appears logged in/out unexpectedly
- Settings or analytics reflect an earlier learner

Look at:

- `GET /auth/current-user`
- Browser cookies/session state
- Current `MIRALINGO_DATABASE_PATH`

Suggested action:

- Log out, reload, or clear session cookies; use a throwaway database path for clean repros.

## Load profile

- Intended load: one local learner in one browser session
- Shared resources:
  - local SQLite database
  - one browser session/cookie jar
  - optional public Iconify API calls
- Audio uses local runtime dependencies and browser playback
- This UAT does not claim production concurrency behavior

## Negative tests checklist

Run and record these negative cases during UAT:

- Wrong typed answer
- Blank typed answer
- `Give Up` reveal path
- Unauthenticated reload/session check after logout or deletion
- Invalid or missing Iconify result fallback/offline behavior
- Deletion confirmation mismatch before successful deletion
